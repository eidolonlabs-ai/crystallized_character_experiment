# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Experiment E002 — "crystallized character" models. LoRA-fine-tunes small open-source LLMs on each character's real conversation history, with synthetic augmentation, so the emergent voice gets baked into the weights rather than relying on a system prompt.

- **Stack**: MLX (`mlx-lm`) exclusively on Apple Silicon. No PyTorch/Docker.
- **Characters**: `baseline` (Lyra Moonwhisper).
- **Models** (the stable IDs used everywhere in scripts): Mistral family (`mistral_v0_3`/`mistral_v0_2`/`mistral_v0_1`) and Llama family (`llama31_8b`/`llama3_8b`/`llama2_7b`). `mistral` is a shorthand alias for `mistral_v0_3`. Each maps to a specific HF repo via `get_hf_model()` in `scripts/model_config.sh`.
- **Variants**: `standard` (8 LoRA layers, 5e-5 lr, 512 seq, 5 epochs) and `deep` (16 LoRA layers, 2.5e-5 lr, 768 seq, 5 epochs). Add `_deep` to output paths.

See `README.md` and `E002_crystallized_character.md` for background/hypothesis, `ARCHITECTURE.md` for pipeline design, and `DATA_PIPELINE.md`, `MLX_SETUP.md`, `A_B_TEST_COMMANDS.md` for deeper reference.

## Environment

```bash
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt   # transformers, mlx-lm, langchain-openai, langchain-core, python-dotenv
# Optional: export OPENROUTER_API_KEY (or LLM_API_KEY) for scripts/generate_synthetic_data.py
```

All training/chat scripts require macOS Apple Silicon and activate `.venv` themselves — fail fast with a clear message if it's missing.

## Common Commands

```bash
# Train one model (standard variant, FP16 by default)
./scripts/train_character_model.sh baseline mistral

# Deep variant + 4-bit fused output
./scripts/train_character_model.sh baseline mistral deep 4bit

# Full baseline sweep across all 6 supported models (~1.5–2 hrs)
./train_baseline_suite.sh test          # dry-run printout
./train_baseline_suite.sh run           # actual training, per-model timing

# Chat interactively with a trained model (uses LoRA adapter + quantized base)
./chat_character.sh baseline mistral
./chat_character.sh baseline mistral deep

# Regression-style check across all trained adapters for one character
./test_trained_models.sh baseline 5

# Export a fused MLX model to LM Studio (~/.lmstudio/models/cynthia/…)
./scripts/export_to_lmstudio.sh baseline mistral

# Direct mlx_lm chat (no wrapper, useful for prompt experiments)
python -m mlx_lm chat --model models/mistral-7b-instruct-v0.3-4bit \
    --adapter-path adapters/baseline_mistral_qlora
```

## Data Pipeline (prepare once, then commit)

Training reads **pre-prepared**, **versioned** datasets from `raw_data/prepared_data/{character}/{variant}_split_<seq>/` — it never re-splits or re-truncates on each run. This is the key reproducibility invariant.

```bash
./prepare_all_datasets.sh                # only when source JSONL changes
git add raw_data/prepared_data/ && git commit
./scripts/train_character_model.sh baseline mistral   # uses prepared data
```

- Source files live at repo root: `raw_data/training_data_<character>{,_augmented,_full}.jsonl`. The `_augmented_curated` variant is generated by running `scripts/curate_training_data.py`.
- Dataset variant priority at training time: `augmented_curated` → `augmented` → `base` → `full` (the script picks the first existing directory matching the model's `MAX_SEQ_LENGTH`). The `augmented_curated` variant is generated by `prepare_all_datasets.sh`, which runs `scripts/curate_training_data.py` to derive it from the augmented set before the split step. If you skip the prep script and only have the legacy source JSONLs, training will silently fall back to `augmented`.
- 90/10 train/valid split is fixed with `random.seed(42)`; pre-truncation keeps system prompt + recent context, trimming only assistant responses.
- `scripts/split_training_data.py` and `scripts/truncate_training_data.py` are the underlying utilities; in normal use only `prepare_all_datasets.sh` calls them.

## Architecture & Layout

```
scripts/train_character_model.sh   # entrypoint: (character, model, variant, quantize)
scripts/train_mlx.py               # preflight checks + shells out to `python -m mlx_lm lora`

test_trained_models.sh             # smoke-tests adapters with tiered prompts per character
scripts/export_to_lmstudio.sh      # copies fused model into ~/.lmstudio/models/cynthia/

raw_data/                          # all JSONL (source + prepared_data/* in git)
adapters/<character>_<model>_qlora[_deep]/         # LoRA weights (gitignored)
models/<character>_<model>_mlx_q4[_deep]/          # fused+quantized MLX (gitignored)
```

## Key Conventions & Gotchas

- **Characters, models, and variants are validated by string match** in both training and chat scripts. Adding a new model requires editing the `case` statement in `get_model_config`/`get_quantized_model_path` of **both** `scripts/train_character_model.sh` and `chat_character.sh`. The same applies to `test_trained_models.sh`'s quantized-model mapping.
- **System prompts in `chat_character.sh` and `scripts/character_config.sh` are hand-written strings** inside `get_system_prompt()`. They are **not** read from the training data. If the training data's system prompt drifts, mirror it in both files before retraining.
- **LoRA path layout is fixed**: `adapters/{character}_{model_name}_qlora[_deep]/adapters.safetensors` (+ `adapter_config.json`). Test/chat scripts pattern-match on this; changing the format will break them.
- **Training is always run with `--no-mask-prompt`** (`MASK_PROMPT="--no-mask-prompt"` in `train_character_model.sh`). Loss is computed on the full sequence including the system prompt. Caveat: every training example in the current corpus has an *identical* ~50-token system prompt, so a non-trivial fraction of every gradient step goes toward re-memorizing that string rather than learning response style. This is a deliberate trade-off — `MLX_SETUP.md` notes that `--mask-prompt` causes NaN loss with Llama 3.1 in current mlx-lm, and the truncate logic in `truncate_training_data.py` already preserves the system prompt unconditionally, so masking would only suppress signal the model will inevitably see anyway.
- **Quantization suffix on fused output**: `_fp16` for `full`, `_q4` for `4bit`. `chat_character.sh` resolves paths via `get_quantized_model_path()` from `scripts/model_config.sh`. The "local quantized" paths in that config (e.g. `models/mistral-7b-instruct-v0.3-4bit`) are a performance optimization — they let mlx-lm skip the on-demand 4-bit conversion. In practice those directories usually don't exist and chat falls back to the HF repo, which mlx-lm auto-downloads and quantizes on first use.
- **Llama 2 chat uses bf16, not the quantized 4-bit or HF fp16.** `chat_character.sh` (and `test_trained_models.sh`) check `models/llama-2-7b-chat-bf16` before falling back; without this, llama2 chat would re-trigger the fp16 activation-overflow instability that the training script already guards against. The bf16 directory is created by `train_character_model.sh` the first time you train llama2.
- **Checkpoints**: `train_mlx.py` auto-detects `[0-9]*_adapters.safetensors` in the adapter dir and resumes via `--resume-adapter-file`. Saves every 50 iters (`--save-every 50`). Don't manually delete intermediate checkpoints without expecting to lose resume capability.
- **Memory tuning**: For machines with <32 GB unified memory, follow the order in `MLX_SETUP.md` → reduce `--gradient-accumulation-steps 1`, then `--max-seq-length 768`, then `--num-layers 8`.
- **Metal/MLX env vars** are exported by both wrappers: `MLX_METAL_DEBUG=1`, `OMP_NUM_THREADS=1`, `MKL_NUM_THREADS=1`. Required for stable Apple Silicon training.
- **JSONL validation**: `scripts/validate_all_jsonl.py` walks every `*.jsonl` under `raw_data/` and flags comment lines or non-JSON content — run this after manual edits; failures abort with all errors.

## Testing & Evaluation

- Smoke-test inference for every trained adapter: `./test_trained_models.sh <character> [num_prompts]`. Pass criterion is ≥50% of prompts producing >10 chars of output within 30s.
- A/B-comparison prompts and hypotheses live in `A_B_TEST_COMMANDS.md`. Use the same prompt set on `standard` vs `deep` adapters and watch for personality depth vs. template repetition.
- `scripts/test_inference_quality.py` is a per-adapter smoke test (pass/fail per model).
- `scripts/evaluate_character.py` is the quantitative eval: runs the same fixed prompt set through every trained adapter and reports response length plus key-phrase hit rates for name/setting/blessings/archaic markers. Use it to compare standard-vs-deep and model-vs-model with numbers, not vibes. Outputs a Markdown table; `--output PATH` dumps per-prompt JSONL for diffing.
